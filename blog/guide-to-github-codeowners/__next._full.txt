12:T653,<p>Ive recently been working in a monorepo environemnt with at present 166 engineers working across a single solution containing a plethora of apps.</p>
<p>We make use of <a href="https://nx.dev/">Nx</a> to achieve this which is a great mono repo framework that works across all the core languages. In its essence it mandates a folder structure comprising of apps and libraries,</p>
<ul>
<li>apps<ul>
<li>apps1</li>
<li>apps2</li>
</ul>
</li>
<li>libs<ul>
<li>library1</li>
<li>library2</li>
</ul>
</li>
</ul>
<p>As you can imagine code ownership and attributing a change to the right teams becomes key. This is where gits <code>CODEOWNERS</code> file comes in, it allows you to use the same glob patterns you use with a <code>.gitignore</code> file to attribute files and folders to a particular team or individual.</p>
<p>Below is an example of what it looks like, generic rules should be placed at the top and specific rules should be at the bottom.</p>
