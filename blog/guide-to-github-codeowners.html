Ive recently been working in a monorepo environemnt with at present 166 engineers working across a single solution containing a plethora of apps.

We make use of Nx to achieve this which is a great mono repo framework that works across all the core languages. In its essence it mandates a folder structure comprising of apps and libraries,

  • apps
    • apps1
    • apps2
  • libs
    • library1
    • library2

As you can imagine code ownership and attributing a change to the right teams becomes key. This is where gits CODEOWNERS file comes in, it allows you to use the same glob patterns you use with a .gitignore file to attribute files and folders to a particular team or individual.

Below is an example of what it looks like, generic rules should be placed at the top and specific rules should be at the bottom.

1/**/* @redteam1 @blueteam2
2/red/* @redteam1
3/blue/* @blueteam1
4/project.json @platformteam
